# FCM 등장 배경과 기술적 한계

> 작성일: 2026-03-20
> 목적: FCM 이전의 모바일 푸시 알림 문제점, FCM이 해결한 문제, 그리고 FCM 자체의 기술적 한계를 체계적으로 정리한다.

---

## 1. FCM 이전: 모바일 푸시 알림의 문제점

### 1.1 Polling 방식의 시대

스마트폰 초기(2008~2010)에는 서버로부터 새로운 데이터를 받기 위해 **클라이언트가 주기적으로 서버에 요청**하는 Polling 방식이 주류였다.

```
Client ──HTTP GET──▶ Server   (매 5분마다 반복)
Client ◀──200 OK── Server    (새 데이터 없음)
Client ──HTTP GET──▶ Server   (5분 후 다시)
Client ◀──200 OK── Server    (새 데이터 있음!)
```

#### Polling의 근본적 문제

| 문제 | 설명 |
|------|------|
| **배터리 소모** | 주기적으로 라디오를 깨워야 하므로 배터리 소모가 극심. 3G 환경에서 라디오 웨이크업 1회 = CPU 활성 상태 수십 초 유지 |
| **실시간성 부재** | Polling 주기(예: 5분)만큼 지연 발생. 1분 주기로 줄이면 배터리가 수시간 내 방전 |
| **서버 부하** | 수백만 디바이스가 동시에 Polling하면 서버에 불필요한 트래픽 폭증. 대부분의 요청은 "새 데이터 없음" 응답 |
| **네트워크 낭비** | 새 데이터가 없어도 매번 TCP 연결 수립 → HTTP 요청 → 응답 사이클 발생 |
| **확장성 한계** | 사용자 수 증가에 비례하여 서버 리소스가 선형 증가 |

### 1.2 Long Polling / Persistent Connection 시도

Polling의 한계를 극복하기 위해 **Long Polling**(서버가 새 데이터가 생길 때까지 응답을 지연) 또는 **지속 연결**(TCP/WebSocket)을 유지하는 방식이 시도되었다.

```
Client ──HTTP GET──▶ Server   (연결 유지, 응답 대기)
         ... 대기 ...
Client ◀──200 OK── Server    (새 데이터 발생 시 즉시 응답)
Client ──HTTP GET──▶ Server   (다시 연결)
```

#### Long Polling / 지속 연결의 문제

| 문제 | 설명 |
|------|------|
| **앱별 개별 연결** | 설치된 앱마다 독립적인 지속 연결 필요. 30개 앱이 각각 연결을 유지하면 30개 TCP 소켓이 동시 활성 |
| **배터리 소모 (여전히)** | 각 연결의 Keep-alive 패킷이 라디오를 반복적으로 깨움 |
| **NAT 타임아웃** | 모바일 통신사의 NAT 테이블이 일정 시간(보통 5~30분) 후 연결을 끊음. 재연결 오버헤드 발생 |
| **개발 복잡도** | 앱 개발자가 직접 연결 관리, 재연결 로직, 인증, 오류 처리를 구현해야 함 |
| **OS 제한** | 모바일 OS가 백그라운드 프로세스와 네트워크 연결을 적극적으로 종료 |

### 1.3 C2DM (Cloud to Device Messaging) — FCM의 전신

Google은 2010년 Android 2.2(Froyo)와 함께 **C2DM(Cloud to Device Messaging)** 을 출시하여 최초의 플랫폼 수준 푸시 인프라를 제공했다.

#### C2DM의 구조

```
App Server ──HTTP POST──▶ C2DM Server ──▶ Android Device
                (API Key 인증)           (Google Play Services 경유)
```

#### C2DM의 한계

| 한계 | 설명 |
|------|------|
| **할당량 제한** | 앱당 일일 전송 건수에 엄격한 제한. 대규모 서비스 운영 불가 |
| **메시지 크기 제한** | 페이로드 최대 **1KB** — 유의미한 데이터 전송이 어려움 |
| **Android 전용** | iOS, Web을 지원하지 않아 플랫폼별 개별 구현 필요 |
| **단순한 API** | 토픽 메시징, 그룹 전송 등 고급 기능 미지원 |
| **신뢰성 부족** | 전달 상태 추적, 오류 진단 도구 미비 |
| **Google 계정 필수** | 디바이스에 Google 계정 로그인이 필요 |

### 1.4 GCM (Google Cloud Messaging) — C2DM의 후속

2012년 Google은 C2DM을 대체하는 **GCM(Google Cloud Messaging)** 을 출시했다.

#### GCM이 개선한 점

- 페이로드 크기 **4KB**로 확대
- 멀티캐스트 메시징 (최대 1,000개 디바이스에 동시 전송)
- 고정 API 키(Server Key) 기반 인증
- 토픽 메시징 지원 추가
- 더 높은 전송 할당량

#### GCM에 여전히 남아 있던 문제

| 문제 | 설명 |
|------|------|
| **고정 API 키 보안 취약** | Server Key가 노출되면 누구나 메시지를 전송 가능. 키 순환(rotation) 메커니즘 부재 |
| **크로스플랫폼 미완성** | iOS 지원이 추가되었지만 별도의 APNs 설정이 필요하고 API가 통합되지 않음 |
| **Legacy HTTP 프로토콜** | JSON 기반이지만 REST 표준에 미달. 에러 응답 형식이 비일관적 |
| **분석 도구 부재** | 전달률, 오류율 등을 모니터링할 대시보드가 없음 |
| **Firebase 미통합** | 인증, 데이터베이스, 애널리틱스 등 다른 백엔드 서비스와 분리되어 운영 |

## 2. FCM이 해결한 문제

2016년 Google은 Firebase 플랫폼에 GCM을 통합하여 **FCM(Firebase Cloud Messaging)** 을 출시했다. FCM은 이전 세대의 핵심 문제들을 다음과 같이 해결했다.

### 2.1 단일 연결로 배터리 문제 해결

**FCM 이전**: 앱마다 개별 서버 연결 → 30개 앱 = 30개 TCP 연결 유지 → 배터리 급속 소모

**FCM 이후**: OS 레벨의 **단일 persistent 연결**을 통해 모든 앱의 푸시를 수신

```
[Before]
App A ──TCP──▶ Server A
App B ──TCP──▶ Server B       ← 앱 수만큼 연결 증가
App C ──TCP──▶ Server C

[After - FCM]
App A ┐
App B ├──단일 연결──▶ FCM Server    ← 연결 1개로 통합
App C ┘
```

- Android에서는 GMS(Google Mobile Services)가 FCM 서버와 **단일 TCP/MQTT 연결**을 유지
- 모든 앱의 푸시 메시지가 이 하나의 연결을 통해 전달
- Keep-alive 주기를 OS가 최적화하여 배터리 소모 최소화

### 2.2 크로스플랫폼 통합 API

**FCM 이전**: Android는 GCM, iOS는 APNs, Web은 별도 구현 → 플랫폼별 3벌 코드 유지

**FCM 이후**: **HTTP v1 API** 하나로 Android, iOS, Web 모두 전송

```json
// 단일 API로 플랫폼별 커스터마이징까지 가능
{
  "message": {
    "topic": "news",
    "notification": { "title": "Breaking", "body": "..." },
    "android": { "priority": "high" },
    "apns": { "headers": { "apns-priority": "10" } },
    "webpush": { "headers": { "Urgency": "high" } }
  }
}
```

### 2.3 보안 강화 (OAuth2)

**GCM**: 고정 Server Key가 노출되면 무제한 메시지 전송 가능, 키 순환 불가

**FCM HTTP v1 API**: **OAuth2 단기 토큰**(~1시간 만료) 사용

| 항목 | GCM (Legacy) | FCM (HTTP v1) |
|------|-------------|---------------|
| 인증 방식 | 고정 Server Key | OAuth2 Access Token |
| 토큰 수명 | 무기한 (수동 폐기만 가능) | ~1시간 (자동 만료) |
| 노출 시 위험 | 무제한 메시지 전송 가능 | 최대 1시간 내 악용, 자동 만료 |
| 키 순환 | 수동, 미지원에 가까움 | 서비스 계정 키 교체 가능 |

### 2.4 Firebase 생태계 통합

FCM은 단독 서비스가 아니라 Firebase 플랫폼의 일부로 동작하면서 다음과 같은 통합 이점을 제공한다:

| 통합 서비스 | FCM과의 연동 |
|-----------|------------|
| **Firebase Analytics** | 알림 전달·열람·클릭 자동 추적 (notifications_received, notification_open 이벤트) |
| **Remote Config** | 알림 내용을 서버 배포 없이 동적 변경 |
| **A/B Testing** | 알림 문구·시간·타겟 실험을 콘솔에서 직접 수행 |
| **Cloud Functions** | Firestore/RTDB 트리거로 자동 알림 발송 |
| **Crashlytics** | 알림 수신 시 크래시 연관 분석 |

### 2.5 서버 부하 및 확장성 문제 해결

**Polling 시대**: N명 사용자 × M분 간격 = N/M 요청/초가 서버에 상시 발생

**FCM 이후**: 서버는 **새 데이터가 있을 때만** FCM API를 호출. 나머지 연결 관리는 Google 인프라가 담당.

- 프로젝트당 **600,000건/분** 전송 가능
- 토픽 메시징으로 서버가 개별 토큰을 관리하지 않아도 브로드캐스트 가능
- 오프라인 디바이스에 대해 최대 **28일간** 메시지 저장 (Store-and-Forward)

## 3. FCM의 기술적 한계

FCM이 해결한 문제에도 불구하고, FCM 자체에 구조적인 기술적 한계가 존재한다.

### 3.1 전달 보장 없음 (Best-Effort Delivery)

**FCM은 어떤 전달 시맨틱(at-most-once, at-least-once, exactly-once)도 공식적으로 보장하지 않는다.**

```
App Server ──POST──▶ FCM    ──?──▶ Device
              │                      │
         HTTP 200 수신            수신 여부 불명
         (= FCM이 수락함)         (= 전달 확인 없음)
```

- `HTTP 200 + message_id`는 "FCM 서버가 메시지를 수락했다"는 의미이지, 디바이스 도달을 의미하지 않음
- MQTT QoS 0 수준 — 수신 확인(ACK) 메커니즘이 프로토콜 레벨에서 존재하지 않음
- FCM 공식 전달률 ~99%는 **서버 수락 기준**이며, 실제 디바이스 렌더링 기준 산업 평균은 **14~48%**

### 3.2 수신 확인(ACK) 메커니즘 부재

FCM은 메시지가 디바이스에 도달했는지, 사용자에게 표시되었는지, 사용자가 확인했는지를 서버에 알려주는 메커니즘을 **제공하지 않는다**.

| 비교 대상 | 수신 확인 방식 |
|----------|-------------|
| **FCM** | 없음 — 서버는 전달 여부를 알 수 없음 |
| **APNs** | 없음 — FCM과 동일하게 수신 확인 미제공 |
| **WhatsApp** | 더블 체크마크 — 서버 도달(✓), 디바이스 수신(✓✓), 읽음(✓✓ 파란색) |
| **MQTT QoS 1** | PUBACK — 브로커가 수신 확인을 클라이언트로부터 받음 |
| **MQTT QoS 2** | 4단계 핸드셰이크 — Exactly-once 보장 |

→ **FCM 위에 App-Level ACK을 직접 구현해야 한다.**

### 3.3 Android OEM 배터리 최적화에 의한 전달 차단

**FCM 신뢰성을 가장 크게 훼손하는 요인.** 시스템 수준에서 발생하며 앱 개발자가 제어할 수 없다.

| OEM | 간섭 메커니즘 | 결과 |
|-----|-------------|------|
| **Xiaomi (MIUI)** | 자체 배터리 최적화가 GMS 프로세스를 강제 종료 | FCM 연결 자체가 끊김 → 전달률 **20~40%** |
| **Huawei (EMUI)** | 앱 자동 실행 차단 + 배터리 최적화 | 전달률 **20~35%** |
| **Samsung (OneUI)** | Sleeping Apps 기능으로 백그라운드 활동 제한 | 전달률 **51~55%** |
| **OnePlus (OxygenOS)** | 공격적 백그라운드 프로세스 킬 | 전달률 **40~55%** |

- 전 세계 Android 기기의 약 **70%가 중국 OEM** (Xiaomi, Huawei, Oppo, Vivo 등)
- 사용자가 직접 앱을 배터리 최적화 화이트리스트에 등록해야만 해결 가능
- **앱 개발자가 프로그래밍으로 해결할 수 없는 구조적 한계**

### 3.4 iOS 플랫폼 제약

| 제약 | 설명 |
|------|------|
| **Silent Push 스로틀링** | Apple이 의도적으로 시간당 **2~3건**으로 제한. 우회 불가 |
| **APNs 프록시 의존** | FCM이 직접 iOS에 전달하지 않고 APNs를 경유 → Apple 인프라 장애 시 FCM도 영향 |
| **Force Kill 시 Silent Push 드롭** | 사용자가 앱을 강제 종료하면 Silent Push가 **100% 드롭** |
| **Background App Refresh 비활성화** | 사용자가 끄면 데이터 갱신 불가 |

### 3.5 메시지 저장 및 순서 보장 한계

| 한계 | 상세 |
|------|------|
| **Non-collapsible 100건 초과 시 전체 삭제** | 디바이스에 대기 중인 축소 불가 메시지가 100건을 넘으면 **전체가 삭제**됨 — 가장 치명적 |
| **메시지 순서 비보장** | FCM은 메시지 전송 순서를 보장하지 않음. 먼저 보낸 메시지가 나중에 도착할 수 있음 |
| **중복 제거 없음** | 동일 메시지가 여러 번 전달될 수 있으며, FCM 자체적으로 중복을 감지하거나 제거하지 않음 |
| **오프라인 저장 기한** | 최대 **28일** — 이후 메시지는 영구 폐기 |
| **Collapsible 버스트 제한** | 디바이스당 **20건**, 리필 속도 **3분/1건** — 빈번한 업데이트 시 이전 메시지가 덮어씌워짐 |

### 3.6 페이로드 크기 제한

| 항목 | 제한 |
|------|------|
| 최대 페이로드 크기 | **4KB** |
| 토픽 메시지 페이로드 | **4KB** (동일) |

- 이미지, 파일 등 대용량 데이터를 직접 전달할 수 없음
- 알림에 포함할 데이터가 4KB를 초과하면 별도 API 호출로 데이터를 가져와야 함
- C2DM의 1KB에서 개선되었지만, 풍부한 콘텐츠 전달에는 여전히 부족

### 3.7 GMS(Google Mobile Services) 의존성

- FCM은 Android에서 **Google Play Services(GMS)** 가 설치되어 있어야 동작
- GMS가 없는 디바이스에서는 FCM이 **완전히 작동하지 않음** (전달률 0%)
- 해당 디바이스: Huawei 신규 기기(2019년 이후), Amazon Fire 태블릿, 일부 중국 내수 기기
- 대안: Huawei Push Kit(HMS), 자체 WebSocket, 또는 SMS Fallback

### 3.8 종단 간 암호화(E2E) 미지원

- FCM은 **전송 중 암호화(TLS)** 만 제공
- FCM 서버는 메시지 페이로드를 **복호화하여 읽을 수 있음**
- 의료, 금융 등 민감 데이터를 다루는 서비스는 **앱 레벨에서 별도 암호화**가 필수
- Google이 메시지 내용에 접근 가능하다는 것 자체가 규제(GDPR, HIPAA 등) 이슈

### 3.9 전달 지연 (Latency) 문제

| 조건 | 지연 시간 |
|------|----------|
| 온라인 + Foreground | **<500ms** (API 응답 기준) |
| Doze 모드 진입 | **15분~수시간** (Maintenance Window까지 대기) |
| OEM 배터리 최적화 활성 | **수분~수시간** (GMS 프로세스 복구까지) |
| 오프라인 → 온라인 복귀 | **즉시~수분** (FCM 재연결 시간) |
| 대규모 팬아웃 (100만+ 디바이스) | **수초~수분** (내부 큐잉 시간) |

- FCM 공식 수치(95%가 350ms 이내)는 **API 응답 시간**이지 **디바이스 도달 시간**이 아님
- 실시간 채팅, 거래 알림 등 지연에 민감한 서비스에는 FCM 단독 사용이 부적합

### 3.10 플랫폼 종속성 (Vendor Lock-in)

- FCM은 **Google의 독점 서비스** — 대안 없이 Google 인프라에 완전히 의존
- Google의 정책 변경, 가격 변경, 서비스 중단 등에 대한 통제권이 없음
- SLA(Service Level Agreement)가 공식적으로 제공되지 않음 — 장애 시 보상이나 책임 없음
- Legacy API(GCM) 지원 종료 사례처럼 마이그레이션이 강제될 수 있음

## 4. 시대별 진화 요약

```
2008~2010  Polling 시대
           └─ 문제: 배터리 소모, 실시간성 부재, 서버 부하
                │
2010       C2DM 출시 (Android 2.2)
           └─ 해결: 플랫폼 수준 푸시 인프라 최초 제공
           └─ 한계: 1KB 페이로드, Android 전용, 낮은 할당량
                │
2012       GCM 출시
           └─ 해결: 4KB 페이로드, 멀티캐스트, 토픽 메시징
           └─ 한계: 고정 API 키 보안 취약, 크로스플랫폼 미완성
                │
2016       FCM 출시 (Firebase 통합)
           └─ 해결: OAuth2 보안, 크로스플랫폼 단일 API, Firebase 생태계 통합
           └─ 한계: Best-effort 전달, ACK 부재, OEM 간섭, E2E 암호화 미지원
```

## 5. 결론

### FCM이 해결한 핵심 문제

1. **배터리 문제**: 앱별 개별 연결 → OS 수준 단일 연결로 통합
2. **개발 복잡도**: 플랫폼별 개별 구현 → 단일 HTTP v1 API
3. **보안**: 고정 API 키 → OAuth2 단기 토큰
4. **확장성**: 클라이언트 Polling → 서버 Push (이벤트 기반)
5. **생태계**: 독립 서비스 → Firebase 통합 (Analytics, A/B Testing, Remote Config)

### FCM에 여전히 남아 있는 근본적 한계

1. **전달을 보장하지 않는다** — Best-effort이며 수신 확인이 없다
2. **OEM 간섭을 제어할 수 없다** — 전 세계 Android의 70%에서 전달률 20~55%
3. **iOS는 Apple 정책에 종속된다** — Silent Push 스로틀링, Force Kill 시 드롭
4. **Google 인프라에 완전히 의존한다** — SLA 없음, 정책 변경 리스크

**→ FCM은 "전달 인프라"이지 "전달 보장 시스템"이 아니다. 신뢰성이 필요한 서비스는 FCM 위에 App-Level ACK, 재전송, 멀티채널 Fallback을 직접 구축해야 한다.**
