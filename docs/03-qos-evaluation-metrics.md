# QoS 평가 지표 및 평가 방법론

> 작성일: 2026-03-18
> 목적: FCM 안정/불안정 케이스에서 QoS를 **어떤 지표로, 어떻게 측정하고, 어떻게 평가**하는지를 정의한다.

---

## 1. 평가 지표 정의

### 1.1 핵심 지표 (Primary Metrics)

#### M1. 전송 성공률 (Delivery Success Rate)

| 항목 | 정의 |
|------|------|
| **수식** | `성공 전달 수 / 전체 전송 시도 수 × 100%` |
| **측정 포인트** | (a) FCM 서버 수락률, (b) 디바이스 실제 수신률 — **두 가지를 구분하여 측정** |
| **왜 두 가지인가** | FCM 서버 수락(HTTP 200)과 디바이스 실제 수신은 다르다. FCM 공식 전달률 ~99%는 (a)이고, 실제 렌더링 기준 산업 평균은 14~48%이다 [^1][^2] |
| **측정 방법** | (a) HTTP 응답 코드 집계 (b) 클라이언트 앱에서 수신 콜백 시 서버로 ACK 전송 |
| **평가 기준** | ≥99%: 우수, ≥95%: 양호, ≥90%: 보통, ≥80%: 미흡, <80%: 불량 |

```
[측정 구조]

  App Server ──POST──→ FCM Server ──전달──→ Device
       │                    │                  │
       │   ◄── HTTP 200 ───┘                  │
       │   (a) 서버 수락률                      │
       │                                       │
       │   ◄───── HTTP ACK callback ───────────┘
       │   (b) 디바이스 실제 수신률
```

#### M2. 전송 지연시간 (End-to-End Latency)

| 항목 | 정의 |
|------|------|
| **수식** | `디바이스 수신 시각 - 서버 전송 시각` (밀리초) |
| **측정 포인트** | (a) API 지연: 서버→FCM 응답 시간, (b) E2E 지연: 서버→디바이스 수신 시간 |
| **왜 두 가지인가** | API 지연은 FCM 인프라 성능, E2E 지연은 Doze/OEM/네트워크 영향 포함 |
| **통계량** | P50 (중앙값), P95, P99, 최대값 |
| **시간 동기화** | NTP 동기화 필수 — 서버와 디바이스 간 시계 오차가 지연 측정에 직접 영향 |
| **평가 기준** | P95 기준: <1s 우수, <5s 양호, <30s 보통, <5분 미흡, ≥5분 불량 |

**시간 동기화가 중요한 이유**: 서버 시각과 디바이스 시각이 동기화되지 않으면 E2E 지연시간이 부정확해진다. 모든 테스트 디바이스에 NTP를 설정하고, 측정 시작 전 시계 오차를 기록한다.

#### M3. 메시지 손실률 (Message Loss Rate)

| 항목 | 정의 |
|------|------|
| **수식** | `(전송 성공 - 실제 수신) / 전송 성공 × 100%` |
| **의미** | FCM이 수락했으나 디바이스에 도달하지 못한 비율 |
| **왜 별도 지표인가** | M1의 (a)와 (b)의 차이를 직접 보여주는 지표. FCM의 "best-effort" 특성을 정량화 |
| **측정 방법** | FCM 수락 message_id 목록과 디바이스 ACK 목록을 대조 |
| **평가 기준** | <0.5% 우수, <2% 양호, <5% 보통, <10% 미흡, ≥10% 불량 |

#### M4. 중복 수신률 (Duplicate Rate)

| 항목 | 정의 |
|------|------|
| **수식** | `(총 수신 횟수 - 고유 메시지 수) / 고유 메시지 수 × 100%` |
| **의미** | 동일 메시지가 여러 번 수신된 비율 |
| **왜 측정하나** | FCM은 중복 제거를 보장하지 않는다 [^3]. 네트워크 재시도/재연결 시 중복 가능 |
| **측정 방법** | 각 메시지에 고유 ID 부여, 디바이스에서 수신 ID 집계 |
| **평가 기준** | 0% 우수, <1% 양호, <5% 보통, ≥5% 불량 |

### 1.2 보조 지표 (Secondary Metrics)

#### M5. 복구 시간 (Mean Time To Recovery, MTTR)

| 항목 | 정의 |
|------|------|
| **수식** | `정상 전달 재개 시각 - 장애 발생 시각` |
| **의미** | 불안정 상태에서 안정 상태로 복귀하는 데 걸리는 시간 |
| **적용 케이스** | Rate Limit 초과(UNSTABLE-03), 인증 만료(UNSTABLE-04), 서버 장애(UNSTABLE-07), 네트워크 복구(UNSTABLE-14) |
| **측정 방법** | 장애 감지 시각 기록 → 연속 3회 성공 전송 시각을 복구 시점으로 정의 |
| **평가 기준** | <10s 우수, <1분 양호, <5분 보통, <30분 미흡, ≥30분 불량 |

#### M6. Dead Letter 비율 (DLQ Rate)

| 항목 | 정의 |
|------|------|
| **수식** | `최종 실패(재시도 모두 소진) 수 / 전체 전송 시도 수 × 100%` |
| **의미** | 모든 재시도를 거친 후에도 전달 불가능한 메시지 비율 |
| **측정 방법** | 재시도 정책(최대 N회, exponential backoff) 소진 후 DLQ에 적재된 건수 |
| **평가 기준** | <0.1% 우수, <1% 양호, <5% 보통, ≥5% 불량 |

#### M7. 재시도 횟수 (Retry Count)

| 항목 | 정의 |
|------|------|
| **수식** | `총 재시도 횟수 / 재시도가 발생한 메시지 수` (평균) |
| **의미** | QoS 메커니즘의 효율성 — 재시도가 많을수록 리소스 소비 증가 |
| **측정 방법** | 각 메시지별 시도 횟수 기록 |
| **보조 지표** | 재시도 성공률 = `재시도 후 성공 수 / 재시도 발생 메시지 수`. 개발자 사례에서 97% [^4] |

#### M8. 처리량 (Throughput)

| 항목 | 정의 |
|------|------|
| **수식** | `단위 시간당 성공 전달 수` (msg/sec) |
| **의미** | 시스템의 최대 처리 능력 |
| **측정 방법** | 점진적 부하 증가(ramp-up)하며 성공률 95% 유지 가능한 최대 속도 |
| **관련 제한** | FCM 프로젝트당 600,000건/분 = 10,000건/초 [^5] |

## 2. 평가 방법론

### 2.1 실험 설계 원칙

#### 대조군-비교군 구조 (Controlled Experiment)

```
                    독립변수 (조작)
                         │
   ┌─────────────────────┼─────────────────────┐
   │                     │                      │
   ▼                     ▼                      ▼
대조군 (Stable)    비교군 A (Unstable)    비교군 B (Unstable)
  S1~S5              U01~U04              U05~U15
   │                     │                      │
   ▼                     ▼                      ▼
종속변수 측정        종속변수 측정          종속변수 측정
 M1~M8               M1~M8                M1~M8
   │                     │                      │
   └─────────────────────┼──────────────────────┘
                         │
                    비교 분석
                  (대조군 기준 저하율)
```

#### 통제 변수

실험 간 공정한 비교를 위해 다음을 고정한다:

| 통제 변수 | 고정 값 | 이유 |
|----------|---------|------|
| 메시지 크기 | 1KB (data 페이로드) | 크기에 따른 변수 제거 |
| Firebase 프로젝트 | 전용 테스트 프로젝트 1개 | 프로젝트 간 할당량 차이 제거 |
| FCM API 버전 | HTTP v1 | 레거시 API 변수 제거 |
| 메시지 우선순위 | HIGH (기본), NORMAL (비교용) | 케이스별 명시 |
| TTL | 86,400초 (24시간) | 기본값 28일의 변수 제거 |
| 테스트 시간대 | 평일 오전 10시~오후 6시 (KST) | 트래픽 패턴 통일 |

#### 샘플 크기 결정 근거

통계적 유의성을 확보하기 위한 최소 샘플 크기 계산:

```
조건:
  - 신뢰수준: 95% (z = 1.96)
  - 허용 오차: ±2%
  - 예상 비율: p = 0.5 (최대 분산)

  n = (z² × p × (1-p)) / E²
  n = (1.96² × 0.5 × 0.5) / 0.02²
  n = 2,401

→ 케이스당 최소 2,500건 전송
→ 반복 실험 3회 (재현성 확인)
```

**근거**: 비율 추정의 표본 크기 공식 (Cochran's formula). 99% vs 85% 전달률 차이를 ±2% 정밀도로 감지하기 위함.

### 2.3 각 지표별 구체적 평가 방법

#### M1. 전송 성공률 — 평가 방법

```
Step 1: 서버 수락률 계산
  - HTTP 200 응답 수 / 전체 전송 수 × 100%
  - 에러 코드별 분류: 400(INVALID), 404(UNREGISTERED), 429(QUOTA), 500/503(서버)

Step 2: 디바이스 수신률 계산
  - ACK 수신 수 / HTTP 200 응답 수 × 100%
  - ACK 대기 시간: TTL 기간 (테스트에서는 24시간)

Step 3: 대조군 대비 비교
  - 저하율 = (대조군 수신률 - 비교군 수신률)
  - 예: 대조군 99% - 비교군 55% = 44%p 저하

Step 4: 통계 검정
  - 이항 검정(Binomial test) 또는 카이제곱 검정
  - p-value < 0.05 시 유의미한 차이로 판정
```

#### M2. 전송 지연시간 — 평가 방법

```
Step 1: 원시 데이터 수집
  - API 지연 = fcm_accepted_at - sent_at
  - E2E 지연 = device_received_at - sent_at

Step 2: 분포 분석
  - 히스토그램 생성 (10ms 구간)
  - 백분위수 계산: P50, P95, P99, Max

Step 3: 대조군 대비 비교
  - P95 비율 = 비교군 P95 / 대조군 P95
  - 예: 대조군 P95=300ms, 비교군 P95=4,200ms → 14배 증가

Step 4: 이상치 분석
  - P99 이상 지연 메시지의 원인 분류
  - Doze 영향 / OEM 영향 / 네트워크 영향 구분
```

#### M3. 메시지 손실률 — 평가 방법

```
Step 1: 대조 (Reconciliation)
  - FCM 수락 message_id 집합 (A)
  - 디바이스 ACK message_id 집합 (B)
  - 손실 집합 = A - B (ACK 대기 시간 24시간 경과 후 판정)

Step 2: 손실 원인 분류
  - FCM Data API 조회: droppedDeviceInactive / TTLExpired / TooManyPending 등
  - 미분류 손실: FCM이 추적하지 못하는 영역 (OEM 킬, iOS 스로틀 등)

Step 3: 플랫폼/OEM별 분석
  - 동일 메시지를 Android(Pixel) + Android(Samsung) + iOS에 동시 전송
  - OEM별 손실률 비교
```

#### M4. 중복 수신률 — 평가 방법

```
Step 1: 디바이스 측 수신 로그
  - 수신된 모든 message_id와 수신 횟수 기록
  - 동일 ID 2회 이상 수신 = 중복

Step 2: 중복 발생 컨텍스트 분석
  - 네트워크 전환 시 / 디바이스 재연결 시 / 서버 재시도 시
  - 시간차(첫 수신 ~ 중복 수신 간 간격) 분석
```

#### M5. 복구 시간 — 평가 방법

```
Step 1: 장애 주입
  - 의도적으로 불안정 조건 생성 (Rate Limit 초과, 네트워크 차단 등)

Step 2: 장애 시작 시각 기록
  - 첫 실패 응답 수신 시각 = T_fault

Step 3: 복구 시점 판정
  - "연속 N회(N=3) 정상 전송 성공" 시각 = T_recovery
  - MTTR = T_recovery - T_fault

Step 4: 복구 패턴 분류
  - 즉시 복구 (< 10s): 일시적 에러
  - 점진적 복구 (10s ~ 5분): Rate limit, 인증 갱신
  - 장기 미복구 (> 5분): OEM 킬, 디바이스 오프라인
```

#### M6~M8. 보조 지표 — 평가 방법

```
M6. DLQ 비율:
  - 재시도 정책: 최대 5회, exponential backoff (1s, 2s, 4s, 8s, 16s)
  - 5회 모두 실패 시 DLQ 적재 → 비율 계산

M7. 재시도 횟수:
  - 메시지별 시도 횟수 분포 히스토그램
  - 평균, 중앙값, 최대값 보고

M8. 처리량:
  - Ramp-up 테스트: 100 → 500 → 1,000 → 5,000 → 10,000 msg/sec
  - 각 단계에서 M1(성공률) ≥ 95% 유지하는 최대 속도 = 유효 처리량
```

## 3. 평가 등급 체계

### 3.1 개별 지표 등급

| 등급 | M1 전달률 | M2 지연(P95) | M3 손실률 | M4 중복률 | M5 복구시간 |
|------|----------|-------------|----------|----------|-----------|
| **A (우수)** | ≥ 99% | < 1s | < 0.5% | 0% | < 10s |
| **B (양호)** | ≥ 95% | < 5s | < 2% | < 1% | < 1분 |
| **C (보통)** | ≥ 90% | < 30s | < 5% | < 5% | < 5분 |
| **D (미흡)** | ≥ 80% | < 5분 | < 10% | < 10% | < 30분 |
| **F (불량)** | < 80% | ≥ 5분 | ≥ 10% | ≥ 10% | ≥ 30분 |

### 3.2 종합 QoS 등급

5개 핵심 지표의 가중 평균으로 산출:

| 지표 | 가중치 | 근거 |
|------|--------|------|
| M1. 전달률 | **40%** | 가장 근본적인 QoS 지표 |
| M2. 지연시간 | **25%** | 실시간성이 중요한 푸시 알림 특성 |
| M3. 손실률 | **20%** | FCM의 best-effort 특성에서 핵심 |
| M4. 중복률 | **10%** | 사용자 경험 영향 |
| M5. 복구시간 | **5%** | 장애 대응 능력 |

```
종합 QoS 점수 = (M1_score × 0.4) + (M2_score × 0.25) + (M3_score × 0.2) + (M4_score × 0.1) + (M5_score × 0.05)

점수 매핑: A=5, B=4, C=3, D=2, F=1

종합 등급:
  ≥ 4.5: Grade A (프로덕션 적합)
  ≥ 3.5: Grade B (조건부 적합)
  ≥ 2.5: Grade C (개선 필요)
  ≥ 1.5: Grade D (심각한 개선 필요)
  < 1.5: Grade F (사용 부적합)
```

## 5. 통계적 유의성 검증 방법

### 5.1 비율 비교 (전달률, 손실률)

- **검정 방법**: 이표본 비율 z-검정 (Two-proportion z-test)
- **귀무가설 H₀**: 대조군 전달률 = 비교군 전달률
- **대립가설 H₁**: 대조군 전달률 ≠ 비교군 전달률
- **유의수준**: α = 0.05
- **효과 크기**: Cohen's h로 실질적 유의성 판단 (h ≥ 0.2: 작은, ≥ 0.5: 중간, ≥ 0.8: 큰 효과)

### 5.2 지연시간 비교

- **검정 방법**: Mann-Whitney U 검정 (비정규 분포 예상)
- **이유**: 지연시간 분포는 일반적으로 우측 꼬리가 긴 비정규 분포
- **보고**: 중앙값 차이 + 효과 크기 (r = Z/√N)

### 5.3 반복 실험 재현성

- 각 케이스 3회 반복 실험
- 반복 간 변동 계수(CV) < 10% 확인
- CV ≥ 10% 시 추가 반복 또는 원인 조사
